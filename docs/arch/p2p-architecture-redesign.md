# 网络架构重构方案（EasyTier 中继 + 局域网直连）

> **文档版本**: v1.9
> **创建日期**: 2026-02-28
> **最后更新**: 2026-02-28
> **状态**: 方案确定，待实施

---

## 1. 背景与问题

当前 N.E.K.O. 使用 FRP 实现穿透，所有通信都绕行公网服务器：

```
手机 ──→ 云服务器(FRP) ──→ 电脑(localhost:48911)
```

核心痛点：带宽成本高、延迟高、服务器单点故障。

### 根本限制：主服务只能绑定 localhost

主服务只能监听 `localhost:48911`，无法改为 `0.0.0.0`。FRP 的 frpc 能工作，是因为它运行在电脑本地，可以读到 localhost，手机本身做不到这一点。

但 frpc 本身就是一个由 `launcher.py` 管理的旁路进程。**同样的模式也可以用来做局域网代理**，主服务完全不需要改动。

---

## 2. 为什么 EasyTier P2P 穿透不适用（但 ZeroTier 可以）

EasyTier 通过创建虚拟网络（`10.x.x.x`）实现设备间通信，只有运行 easytier-core 的设备才能加入。EasyTier **没有官方移动端 App 或 SDK**，手机无法加入虚拟网络，所以方案失效。

ZeroTier 原理相同，但有官方 iOS/Android App，手机可以加入虚拟网络。**这是两者的根本区别。**

---

## 3. 结论：EasyTier 对本项目的实际价值非常有限

综合以上两点：

| 场景 | FRP | EasyTier | 说明 |
|------|-----|----------|------|
| 局域网直连 | ❌ 不支持 | ❌ 不支持 | 主服务绑定 localhost，任何方案都无法绕过 |
| 跨网 P2P | ❌ 不支持 | ❌ 不支持 | 手机无 EasyTier SDK，无法加入虚拟网络 |
| 跨网中继 | ✅ 支持 | ✅ 支持 | 两者功能等价，EasyTier 稍复杂 |

**将 FRP 替换为 EasyTier 不会带来任何用户体验上的改善**，只是换了一个中继协议，服务器带宽消耗和延迟完全持平。考虑到 EasyTier 引入了额外的配置复杂度（每用户需要独立虚拟 IP 和网络名以避免冲突），以及二进制体积增加约 10-20MB，**性价比不高，不建议推进**。

---

## 4. 真正能改善现状的方向

### 方向 A：局域网代理进程（简单，收益最大）

这是最直接的方案。在桌面端随主服务一起启动一个轻量 TCP 代理进程，监听 `0.0.0.0:48920`，把流量转发到 `localhost:48911`。主服务不动，手机只要和电脑在同一个 WiFi 下，就能直接连到这个端口，完全绕过服务器。

```
手机 ──→ 电脑:48920（局域网代理进程）──→ localhost:48911（主服务）
```

管理方式和现有 frpc 完全一样，由 `launcher.py` 启动和销毁。代码量极小（Python asyncio 30-50 行），没有任何外部依赖。

移动端需要配合实现局域网 IP 检测（见 v1.2 中 ConnectionManager 的描述），优先尝试局域网，失败再回退到 FRP 中继。

**收益**：同 WiFi 用户延迟从 100-200ms 降到 10ms 以内，服务器带宽消耗归零。这是本项目最值得做的改动。

---

### 方向 B：跨网场景——没有简单的办法

跨网（手机在 4G/5G，电脑在家）的本质问题是：两台设备在不同的 NAT 后面，互相不可见，**必须有第三方介入**。介入方式只有两种：

- **中继**：流量经过服务器转发，延迟高、成本高，但实现简单。当前 FRP 就是这种。
- **NAT 打洞（P2P）**：两端通过协商直接建立连接，绕过服务器，延迟低，但实现复杂。

目前的实际选项：

| 方案 | 用户体验 | 实现复杂度 | 说明 |
|------|---------|-----------|------|
| **继续用 FRP（维持现状）** | 无变化 | 零 | 跨网延迟 100-300ms，持续消耗服务器带宽 |
| **UPnP + 云端协调（轻量 P2P）** | 透明 | 低 | 实现简单，但依赖路由器支持 UPnP，覆盖率约 50-70% |
| **ZeroTier + LAN 代理** | 需安装 ZeroTier App | 低 | 有移动端 App，NAT 打洞内置，实现远比 WebRTC 简单 |
| **WebRTC DataChannel（自研 P2P）** | 透明，用户无感知 | 高 | 打洞成功时延迟 30-80ms，失败自动回退中继 |
| **Tailscale** | 需注册账号 + 安装 App | 低 | 接入简单，但依赖 Tailscale 商业基础设施 |

**FRP 继续用**是最省力的选择——跨网场景本来就是 FRP 的主战场，它做得很好。如果跨网用户占比不高，或者延迟在可接受范围内，就没必要动它。

**UPnP + 云端协调**是性价比最高的 P2P 方案，详见方向 C。

**ZeroTier** 国内可用性有问题，详见方向 D。

**WebRTC 方案**适合对用户体验要求极高且愿意投入大量工程的情况，详见方向 E。

**Tailscale** 则适合内部/企业场景，不太适合面向普通用户的产品。

---

### 方向 C：UPnP + 云端协调（轻量 P2P，优先考虑）

#### 原理

"云服务器辅助 P2P"的最简实现，不需要 WebRTC，不需要自定义帧协议：

1. 桌面端启动时，用 UPnP 请求家用路由器在公网上开一个端口（如 48920），映射到本机的 LAN 代理进程
2. 把获得的公网 IP:端口上报到云服务器（一个简单的 HTTP 接口）
3. 手机发起连接时，先从云服务器查到桌面端的公网 IP:端口
4. 手机直接连接，绕过服务器
5. 连接失败（UPnP 不可用 / 对称 NAT）时，回退到 FRP

```
桌面端 → UPnP → 路由器开端口 → 公网 IP:48920
                                          ↑
云服务器（只存一条 IP:端口记录）          │ 查询
                                          │
手机 ──────────────────────────────────→ 直连
```

整个方案的新增代码量：
- 桌面端：Python `miniupnpc` 库，几十行，自动做 UPnP 映射并上报云服务器
- 云服务器：一个记录 IP:端口的 HTTP 接口，十几行
- 移动端：先查云服务器拿 IP，尝试直连，失败回退 FRP，约 30-50 行

**这是远比 WebRTC 简单的 P2P 方案，总代码量不超过 200 行。**

#### 覆盖范围与限制

| 场景 | 能否直连 | 说明 |
|------|---------|------|
| 家庭宽带 + UPnP 开启（大多数路由器默认开启） | ✅ | 覆盖绝大多数在家的用户 |
| 家庭宽带 + UPnP 关闭 | ❌ | 回退 FRP |
| 运营商 CGNAT（家庭宽带无公网 IP） | ❌ | 回退 FRP，国内部分地区有此情况 |
| 手机 5G/4G（CGNAT，只发起连接，不接收） | ✅ | 手机作为客户端发起，不受 CGNAT 影响 |

实际覆盖率预计 **50-70%** 的跨网用户可以走直连，其余回退 FRP，用户无感知。

#### 与 WebRTC 的对比

| | UPnP 方案 | WebRTC 方案 |
|--|----------|------------|
| 实现复杂度 | 低（~200 行） | 高（~2000 行） |
| P2P 覆盖率 | 50-70% | 60-75% |
| 依赖 | miniupnpc | aiortc / react-native-webrtc |
| 路由器依赖 | 需要 UPnP | 不需要 |

覆盖率相差不大，但工程量差了 10 倍。**建议先做 UPnP 方案，覆盖大多数用户，若效果不够再考虑 WebRTC。**

---

### 方向 D：ZeroTier + LAN 代理（**国内可用性存疑，谨慎选用**）

#### 国内可用性问题（关键限制）

ZeroTier 的控制平面（ZeroTier Central）和全球中继节点（PLANET）都在境外。在国内使用时：

- **控制平面访问不稳定**：ZeroTier Central 在国内时常无法访问，影响设备注册和网络管理
- **打洞失败时中继极慢**：对称 NAT 回退到 ZeroTier 的全球 PLANET 节点，流量绕行境外，延迟可能高达 500ms+，比 FRP 还差
- **UDP 流量可能被 QoS 限速**：国内部分运营商对非标准 UDP 端口有限速策略

**部分缓解方案：自建 Moon 节点**

ZeroTier 支持在自己的服务器上部署 Moon 节点，作为本地根服务器替代 PLANET。将 Moon 部署在国内云服务器上，可以解决中继延迟问题，但控制平面依然依赖 ZeroTier Central（或需要自建更复杂的网络控制器）。

**结论**：ZeroTier 在国内的可用性不稳定，不适合作为面向普通用户的方案。**国内场景推荐直接用方向 D（WebRTC + 自建 STUN/TURN）**，所有基础设施都在国内服务器上，不依赖任何境外服务。

#### 为什么 ZeroTier 在这里能工作（境外场景参考）

ZeroTier 和 EasyTier 原理相同，区别在于 ZeroTier 有官方 iOS/Android App，用户可以在手机上安装并加入虚拟网络。一旦手机加入了桌面端所在的 ZeroTier 网络，手机就能通过虚拟 IP 访问桌面端的 LAN 代理端口（48920），进而访问主服务。

```
手机（ZeroTier App）
        │
        │ ZeroTier 虚拟网络（NAT 打洞，失败则走 ZeroTier 全球中继）
        │
        ▼
桌面端 ZeroTier 虚拟 IP:48920
        │
        │ LAN 代理进程（方向 A 中已规划）
        ▼
   localhost:48911（主服务，无需改动）
```

LAN 代理进程监听 `0.0.0.0:48920`，自然也包含了 ZeroTier 虚拟网卡的地址。**方向 A（局域网代理）和方向 C 共用同一个代理进程**，不需要额外实现。

#### 需要新增的部分

**桌面端**：在 `launcher.py` 中增加对 `zerotier-one` 二进制的管理（和 frpc 一样的模式）。桌面端启动时加入一个每用户唯一的 ZeroTier 网络，并告知移动端网络 ID。

**移动端**：连接流程需要引导用户：安装 ZeroTier App → 加入网络（扫码或输入网络 ID）→ 之后 N.E.K.O. App 自动连接虚拟 IP。这个引导是一次性的。之后手机的 ConnectionManager 在局域网失败后优先尝试 ZeroTier 虚拟 IP，失败再走 FRP。

#### 限制与风险

| 问题 | 说明 |
|------|------|
| 用户需安装 ZeroTier App | 一次性操作，但有摩擦；对技术不熟悉的用户可能放弃 |
| 依赖 ZeroTier Central 控制平面 | 免费套餐限 10 台设备；ZeroTier 服务故障时无法建立新连接（已有连接不受影响） |
| 对称 NAT 打洞失败 | ZeroTier 回退到其全球中继节点（PLANET），延迟可能较高 |
| 自托管控制器复杂 | ZeroTier 支持自托管，但配置比 FRP 复杂；小规模使用 ZeroTier Central 免费套餐即可 |

#### 与 WebRTC 的选择依据

- 如果**用户摩擦可接受**（愿意引导用户装一个 App）→ ZeroTier，实现量少得多
- 如果**用户体验必须透明**（不能让用户感知到额外步骤）→ WebRTC DataChannel

---

### 方向 E：WebRTC DataChannel P2P

#### 基本原理

WebRTC DataChannel 用于解决**跨网且不能要求用户安装额外 App** 的场景，也是**国内跨网 P2P 最可靠的方案**——STUN 和 TURN 都可以完全自建在国内服务器上，不依赖任何境外服务。

WebRTC 是浏览器/移动端原生支持的 P2P 协议，它不只是用于音视频，DataChannel 可以传输任意二进制数据。NAT 打洞能力内置在协议里，不需要第三方工具。

关键点在于：WebRTC P2P 通道建立之后，桌面端可以在本地把收到的数据转发给 `localhost:48911`，对主服务完全透明。主服务不需要做任何改动。

```
手机(react-native-webrtc)
        │
        │ WebRTC DataChannel (P2P 直连，打洞成功时无服务器流量)
        │
        ▼
桌面端 WebRTC 代理进程（新增，类似 frpc 的角色）
        │
        │ 本地转发
        ▼
   localhost:48911（主服务，无需改动）
```

打洞失败时（约 20-40% 的情况，主要是运营商 CGNAT），DataChannel 自动切换到 TURN 中继，用户无感知，行为退化为与现有 FRP 等价。

#### 三个必须新增的组件

**1. 信令服务器（部署在云服务器上）**

WebRTC 建立连接前，两端需要交换网络协商信息（SDP 和 ICE candidates）。这个交换过程需要一个中间人，即信令服务器。它只负责转发协商消息，不经过任何用户数据，流量极小。

可以直接加在现有云服务器上，一个 WebSocket 服务即可。桌面端启动后连上去注册，手机发起连接时通过它与桌面端完成握手，然后信令服务器的任务就结束了。

**2. 桌面端 WebRTC 代理进程（新增，随主服务一起启动）**

类似现有的 frpc，作为一个独立进程由 `launcher.py` 管理。职责：

- 启动时连接信令服务器，注册自己的会话 ID（由用户配置派生，手机端据此找到对应的桌面端）
- 与手机端完成 ICE 协商，建立 DataChannel
- DataChannel 建立后，充当本地代理：收到手机发来的 HTTP 请求帧 → 转发给 `localhost:48911` → 把响应打包发回去
- WebSocket 的持久连接也需要通过 DataChannel 多路复用

这部分是整个方案最复杂的地方，详见下面的技术难点。

可选技术栈：
- Python `aiortc`（与现有 Python 栈一致，但 CPU 占用稍高）
- Node.js `node-datachannel`（更轻量，但增加了运行时依赖）

**3. 移动端 DataChannel 客户端（修改现有 HTTP/WS 层）**

在 `project-neko-request` 和 `project-neko-realtime` 的底层加一层路由：当 WebRTC 通道可用时，把请求序列化后走 DataChannel，而不是直接发 HTTP/WebSocket。对上层业务逻辑透明。

依赖 `react-native-webrtc`，该库在 iOS/Android 上都有稳定支持。

#### STUN/TURN 服务器

- **STUN**：用于发现公网 IP/端口。可使用自建 STUN（coturn 同时支持 STUN 和 TURN），部署在国内服务器，零成本且无境外依赖。
- **TURN**：打洞失败时的中继，部署在国内云服务器上，只有打洞失败的用户才会用到，带宽消耗与 FRP 等价，延迟与 FRP 相当。

#### NAT 打洞成功率预期

| 网络环境 | 预期成功率 | 说明 |
|---------|-----------|------|
| 家庭宽带（普通锥形 NAT） | 75-85% | 大多数家用路由器 |
| 手机 5G/4G（运营商 CGNAT） | 40-60% | 国内运营商大量使用对称 NAT |
| 综合估计 | 约 60-75% | 取决于用户群体的网络构成 |

打洞失败的用户自动走 TURN 中继，对体验无影响，只是不省流量。

#### 最大的技术难点

**HTTP/WebSocket over DataChannel 的帧协议**

DataChannel 是消息式的（类似 UDP），而 HTTP 是请求-响应式，WebSocket 是持久双向流。需要自己设计一个简单的帧格式来复用同一条 DataChannel，包括：

- 请求 ID（用于匹配响应）
- 消息类型（HTTP 请求、HTTP 响应、WebSocket 帧、控制帧）
- 分包与重组（大响应体需要分片）

这个帧协议需要在桌面代理和移动客户端各实现一遍，且必须处理好连接断开、重连期间的请求状态恢复。这是整个方案中代码量最大、最容易出 bug 的部分。

#### 工作量评估

| 组件 | 工作量 | 风险 |
|------|--------|------|
| 信令服务器 | 小（~200 行 WebSocket 服务） | 低 |
| TURN 服务器部署 | 小（coturn 开源方案） | 低 |
| 桌面端 WebRTC 代理 | 大（帧协议 + HTTP/WS 代理逻辑） | 高 |
| 移动端 DataChannel 客户端 | 中（适配现有请求层） | 中 |
| 整体联调测试 | 大（多种 NAT 组合、断线重连） | 高 |

总体来说是一个**工程量不小、但技术上完全可行**的方案。核心价值是让 60-75% 的跨网用户获得真正的 P2P 体验，延迟从 100-300ms 降到 30-80ms，且不依赖任何第三方商业服务。
